## 1. 一句話定位

**一座每天自動出一款 HTML5 小遊戲的「會學習的遊戲工作室」**——目標不是量產，而是透過「解構熱門遊戲 → 帶理論開發 → 出廠自評 → 玩家回饋 → 每週檢討」的閉環，讓**每一款比上一款更好玩**。

靈感原型：致敬當年的「史萊姆好玩遊戲區」，主角美術統一為「史萊姆貓」宇宙（綠色史萊姆＋貓耳）。

## 2. 核心假設（歡迎挑戰）

1. **好玩是可以拆解的**：熱門遊戲的上癮機制（near-miss、歸因於己、指數獎勵、損失趨避）可以被系統性解構成可複用的設計原則。
2. **AI＋設計理論 > AI 裸生成**：給生成器一份不斷成長的「設計聖經」＋當次解構筆記＋歷史教訓，品質會顯著高於單純叫 AI「做一款好玩的遊戲」。
3. **回饋層級**：玩家真實評分 ＞ AI 自評 ＞ 設計理論。三者衝突時以左邊為準。
4. **小步快跑**：單檔 HTML5、30 秒上手、一局 1~3 分鐘的超輕量遊戲，是「每日一款＋快速驗證假設」的合理載體。

## 4. 每日生產線（五階段細節）

### ① 解構（策劃課）
- 從 App Store 台灣免費遊戲榜 Top 100 挑一款「核心玩法能濃縮成 30 秒上手小遊戲」的作品（避開博弈、重度 RPG、純 IP 作）。
- 產出**解構筆記**，格式固定六節：核心迴圈（一圈幾秒）／上癮機制（心理學拆解）／難度與節奏／可偷的設計／不可行的部分／我們的變形版一句話企劃。
- 筆記永久累積——工作室的「策劃資料庫」隨時間變厚。
- 防重複：已用過的靈感與本站已有的核心機制都列入排除清單（**機制多樣性本身也是留存設計**）。

### ② 設計＋實作（開發課）
- 生成器收到三份資料：設計聖經全文＋當次解構筆記＋最近 60 行教訓（含玩家評分）。
- 硬性規格：單一 HTML 檔零外部資源、canvas 400×600 直式、觸控＋鍵盤雙支援、標題畫面／即時分數／最高分／一鍵重開、繁中介面、30 秒上手。
- 出貨檢查清單直接寫進生成指令：**前 15 秒不會死、每個互動都有 juice、Game Over 顯示差 X 分破紀錄、重開零等待**。

### ③ 品管（Playwright 實機測試）
- 無頭瀏覽器實際載入遊戲：console error／未捕捉例外／canvas 存在性檢查＋自動截圖。
- 不通過→帶著錯誤訊息重生一次；再不過→今日停產（寧缺勿濫），失敗原因推播通知。

### ④ 出廠自評（評審課）
- AI 評審以**五維量表**讀碼評分（滿分 50）：
  | 維度 | 檢驗問題 |
  |------|---------|
  | 上手 | 不看說明能玩？規則一句話？ |
  | Juice | 每個操作有視聽回饋？得分有爽感演出？ |
  | 目標 | 玩家隨時知道為何而玩？ |
  | 難度 | 前 15 秒安全？2 分鐘後仍有挑戰？ |
  | 再一局 | 有 near-miss 設計？重開零摩擦？ |
- 分數上架到大廳卡片；「最重要的 3 個改進點」寫進教訓，**餵給下一款**。

### ⑤ 上架
- 登記名錄 → 重建大廳 → git push 自動部署 → Telegram 推新品通知＋截圖。

## 5. 知識庫（工作室的大腦）

| 文件 | 角色 | 更新者 |
|------|------|--------|
| `fun_principles.md` 設計聖經 | 上癮機制理論＋經典解構（2048/Flappy/Block Blast/Suika/割草）＋出貨清單＋評審量表 | 人工初版；**每週檢討可自動修訂**（後設學習） |
| `learnings.md` 教訓集 | 玩家評分、AI 自評改進點、bug 修復根因，一行一條流水帳 | 每次評分/自評/回報/修復自動追加 |
| `deconstructions/` 解構筆記庫 | 每天一份熱門遊戲的上癮機制拆解 | 每日生產線自動累積 |

設計聖經是「慢變數」（理論），教訓集是「快變數」（實證）。生產時教訓權重最高，因為它包含玩家真實反應。

## 6. 數據迴圈（匿名遙測）

- **事件**：`open`（開啟）／`session`（活躍秒數）／`over`（一局結束＋分數）。
- **關鍵設計——活躍時長而非停留時長**：只累積「有互動」的時間，互動間隔超過 30 秒的部分不計。頁面開著掛機不會灌水（實測：玩 10 秒＋掛機 10 分鐘＋玩 5 秒 → 回報 56 秒，而非 615 秒）。
- **指標**：開啟次數／不重複裝置／活躍停留中位數／D1+ 回訪率／分數中位數（中位數抗極端值與偽造）。
- **隱私**：隨機裝置字串、無姓名、IP 不落地、無 cookie。
- **防汙染**：接收端白名單驗證（事件種類/id 格式/數值範圍）＋報表端只認名錄內遊戲 id＋開發者裝置排除開關。
- 天花板（已知且接受）：善意回報制，懂技術的人可偽造合法格式數據；中位數統計讓單點偽造難以撼動整體。

## 7. 維護迴圈（玩家回報 → AI 修復）

> 新遊戲會一直出，但**已上架的遊戲壞了也要有人修**——這是量產型工作室最容易爛尾的地方。

- **回報**：玩家（目前透過 Telegram）一句話回報，例「毛球塞爆了 拖曳方塊會錯位」→ 登記為該遊戲的 open bug，同時寫進教訓集（**新遊戲不再犯同類錯**）。
- **修復**：AI 修復器讀「完整原始碼＋bug 描述」，合約是**最小侵入**——只修 bug，玩法/美術/計分不可動；修完跑同一套 Playwright 品管，**通過才覆寫上線**，失敗保留原版。
- **兩個觸發點**：隨時手動觸發（約 10~20 分鐘）；或每週檢討開場自動掃描所有 open bugs 逐一修。
- 實例（2026-07-04）：《毛球塞爆了》拖放錯位，根因是「視覺以手指為中心畫、判定卻拿手指當方塊左上格」——修復原則「視覺、預覽、判定共用同一個計算函式＋觸控上浮 70px＋round 吸附」已回寫教訓集。

## 8. 每週檢討（後設學習）

每週日 18:00 自動執行，像遊戲公司的週會：

1. 先修完所有玩家回報的 bug（維護債清零再檢討）。
2. 彙整：全部教訓＋每款的玩家評分 vs AI 自評 vs 遊玩數據。
3. AI 製作人回答三個問題：**哪些設計被數據/評分證實有效？哪些假設被打臉？下一款該押什麼方向？**
4. 產出 3~5 條新教訓；若有必要，**直接修訂設計聖經本身**（理論被實證推翻時，改理論）。
5. 週報推播給老闆（我）。

## 9. 品質與安全防線

| 防線 | 內容 |
|------|------|
| 生成安全 | 生成器零工具權限（純文字交稿），不可能動到站台其他部分 |
| 上架品管 | Playwright 實機測試不過不上架；修復同標準 |
| 遙測防灌水 | 活躍時長＋接收端白名單＋數值上限＋中位數統計 |
| 防注入 | 事件 id 只收 `[a-z0-9-]`，公式注入/垃圾字串進不了資料庫與週報 |
| 防塞爆 | 資料庫超過 20 萬筆自動停收（最壞情況=數據停收，站台不受影響） |
| 機密 | 公開 repo 零 token/金鑰（已掃描確認） |
| 回滾 | 全站 git 版控，任何一版遊戲可隨時回滾 |

## 10. 已知限制（誠實面對）

1. **單檔小遊戲的天花板**：無後端、無存檔雲同步、無多人。深度玩法（meta progression、經濟系統）做不了——刻意取捨，聚焦「一局內的樂趣」。
2. **AI 自評是讀碼想像，不是實玩**：五維分數會系統性偏樂觀；靠玩家評分校正。
3. **樣本量**：朋友圈流量小，留存率等指標要累積數週才有統計意義；前期以定性回饋（評語）為主。
4. **美術同質化風險**：全站統一史萊姆貓宇宙是特色也是限制，玩久了視覺疲勞——目前靠「機制多樣性」對沖。
5. **修復迴圈的品管盲區**：煙霧測試抓得到「壞掉」，抓不到「手感變差」——修復後仍需玩家二次確認。
